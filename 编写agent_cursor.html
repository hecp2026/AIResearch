1. Agent 与 LangGraph 到底解决什么

普通 LLM 调用是“一问一答”；Agent 让模型在一个受控循环里反复观察状态、选择工具、执行动作、读取结果，直到得到最终答案。LangGraph 把这个循环显式建模成一张有状态的图，让流程可控、可观察、可恢复。

S

State 状态

在节点间流动的唯一“共享白板”：消息历史、改动的文件、执行过的命令等。

N

Node 节点

一段确定职责的函数：调用模型、执行工具、人工审批、总结或落库。

E

Edge 边

决定下一步去哪里。条件边把“有工具调用”导向工具，把“最终回复”导向结束。

什么时候值得用 LangGraph

场景建议原因
单次摘要、翻译、分类直接调用模型没有循环、状态恢复或工具编排需求。
标准“模型 ↔ 工具”Agent可先用 LangChain create_agent它本身就运行在 LangGraph 上，样板代码更少。
自定义审批、重试、分支、并行或恢复用 LangGraph需要明确控制流程、状态与持久化边界。
代码修改、数据处理、长任务LangGraph + 隔离工具可观察、可恢复、可审计，还能把高风险动作放进审批节点。
版本提醒：LangGraph 1.x 已弃用 langgraph.prebuilt.create_react_agent，高层入口迁移为 langchain.agents.create_agent。本文为讲清底层流程，直接使用稳定的 StateGraph、ToolNode、条件边与 interrupt。

2. 最重要的心智模型：受控的 ReAct 循环

agent读状态，决定回复或调用工具
→
Router检查 tool_calls / 是否需审批
→
tools执行函数，产出 ToolMessage
有工具调用：tools → agent；需审批：先 approval；无工具调用：agent → END

一次 Agent 执行不是模型“在内部神秘地做完所有事”，而是多轮可见的状态转换：

  1. 用户消息进入状态的 messages。
  2. 模型读取系统提示与历史，返回普通文本或结构化 tool_calls。
  3. 路由函数判断：无工具 → 结束；有敏感工具且开启审批 → approval；否则 → tools。
  4. ToolNode 执行对应 Python 函数，把结果作为 ToolMessage 追加回状态。
  5. 模型基于工具结果继续规划，直到给出不含工具调用的最终消息。
  6. 检查点在超步边界保存状态，之后可用同一 thread_id 继续或恢复。
关键原则：模型只负责“决定”，工具负责“执行”，图负责“约束流程”，状态负责“保存事实”。绝不要让模型的自然语言文本本身承担真实世界的副作用。

3. 编写 Agent 的完整流程

把目标写成可验收的任务

先定义输入、输出与完成条件。例如：“运行测试找出失败项，修复实现，使 pytest 全绿，并报告改动与验证结果”。无法客观判断完成，Agent 就难以可靠停止。

划定权限与风险边界

明确可读/可写目录、可运行命令、联网权限、超时、输出上限、敏感文件与必须人工确认的动作。这些边界必须写进工具代码，而不是只写进提示词。

设计状态 Schema

最小 Agent 可只用消息。本例用 TypedDict 额外记录 changed_files、commands_run，并为累积字段配置 reducer，让多次工具调用能“追加”而不是“覆盖”。

把能力做成窄而清晰的工具

每个工具只做一件事，参数类型与 docstring 要准确——模型正是靠它们决定何时调用。对路径、参数、返回长度与异常做校验，把安全逻辑独立成零依赖模块以便单测。

初始化外部模型并绑定工具

密钥来自环境变量。选支持 tool calling 的模型，用 bind_tools 把工具 Schema 交给模型。

实现节点

模型节点返回状态增量 {"messages": [response]}；工具节点用 ToolNode，它负责并行工具调用与 ToolMessage 对齐；需要时增加审批节点。

连边并编译图

从 START 到 agent；agent 经条件边去 tools / approval / END；工具执行后回 agent。编译时注入 checkpointer。

流式运行与可观察性

开发期查看每个状态更新与工具调用；面向用户只输出进度与最终答案。记录延迟、token、工具错误、循环次数与验收结果。

用故障案例验证

至少测试路径穿越、超大文件、命令超时、工具异常、测试失败、审批拒绝、对话恢复与最大循环限制。用假模型驱动图，不消耗 token。

6. 关键实现逐段解析

完整实现见 agent.py 与 sandbox.py。下面拆开最影响正确性与安全性的部分。

6.1 状态：自定义 Schema 与 reducer

最小 Agent 可只用消息。本例还要记录“改了哪些文件”“跑了哪些命令”，于是用 TypedDict 定义状态，并给累积字段配置 reducer：

agent.py · AgentState
import operator
from typing import Annotated
from typing_extensions import TypedDict
from langgraph.graph.message import add_messages

class AgentState(TypedDict):
    messages: Annotated[list, add_messages]        # 追加式消息
    changed_files: Annotated[list[str], operator.add]  # 累积改动文件
    commands_run: Annotated[list[str], operator.add]   # 累积执行命令
    approved: bool                                  # 审批结果（后写覆盖）
Reducer 很关键：同一字段被多个节点写入时，未配置 reducer 通常是“覆盖”。消息、文件清单这类累积值必须显式声明合并方式（add_messages / operator.add），否则后一次工具结果会冲掉前一次。

6.2 工具的真正权限边界：零依赖的 sandbox.py

安全逻辑独立成一个不依赖任何第三方库的模块，这样最关键的部分可以用标准库单测，与模型 SDK 完全解耦。

sandbox.py · 路径限定（节选）
def safe_path(self, relative_path: str) -> Path:
    if not relative_path or "\x00" in relative_path:
        raise WorkspaceError("路径不能为空或包含空字符")
    candidate = (self.root / relative_path).resolve()
    if candidate != self.root and self.root not in candidate.parents:
        raise WorkspaceError("拒绝访问工作区之外的路径")
    if is_sensitive(candidate):
        raise WorkspaceError("拒绝访问可能包含密钥或凭据的文件")
    return candidate
  • 先 resolve() 再检查父目录，可挡住 ../ 与多数符号链接逃逸。
  • 读文件限制大小、编码与返回长度，避免上下文被巨型文件撑满。
  • 搜索限制匹配条数，列表限制文件数。
  • 敏感文件（.env、私钥、凭据）在读写入口都拦截。
  • 命令用参数数组 + shell=False，再校验程序白名单与超时。
白名单 ≠ 沙箱。允许 python 就意味着被运行的脚本仍能执行任意系统调用。真正的生产环境要叠加容器、低权限用户、只挂载临时工作区、网络策略、资源配额与人工审批。

6.3 把 sandbox 封装成工具，并让工具写回状态

只读工具直接返回字符串；有副作用的 write_file、run_command 返回 Command，同时把 ToolMessage 和自定义状态字段一起更新。InjectedToolCallId 由 LangGraph 注入，用于生成配对的 ToolMessage。

agent.py · 会写状态的工具
from langchain_core.tools import tool
from langchain_core.tools.base import InjectedToolCallId
from langchain_core.messages import ToolMessage
from langgraph.types import Command

@tool
def write_file(relative_path: str, content: str,
               tool_call_id: Annotated[str, InjectedToolCallId]) -> Command:
    """将完整 UTF-8 content 写入工作区文件。"""
    message, changed = workspace.write_file(relative_path, content)
    update = {"messages": [ToolMessage(message, tool_call_id=tool_call_id)]}
    if changed is not None:
        update["changed_files"] = [changed]  # reducer 会追加
    return Command(update=update)

这样每成功写一个文件，changed_files 就多一项；run_command 同理把命令记入 commands_run。这些累积事实让 /state 命令与生产审计都变得简单。

6.5 模型节点：输入完整状态，输出状态增量

agent.py · 模型节点
def call_model(state: AgentState):
    response = model_with_tools.invoke([
        SystemMessage(content=SYSTEM_PROMPT),
        *state["messages"],
    ])
    return {"messages": [response]}

系统提示在每次调用时作为首条消息注入，不写入持久化历史。返回列表只含本轮新消息，reducer 负责追加。

6.6 图：条件边 + 审批分支

agent.py · 组图
def route_after_agent(state):
    if tools_condition(state) != "tools":
        return END                     # 无工具调用 → 结束
    last = state["messages"][-1]
    if require_approval and any(tc["name"] in SENSITIVE_TOOLS
                              for tc in last.tool_calls):
        return "approval"            # 敏感工具 → 先审批
    return "tools"

builder = StateGraph(AgentState)
builder.add_node("agent", call_model)
builder.add_node("tools", ToolNode(tools, handle_tool_errors=True))
builder.add_node("approval", approval_node)
builder.add_edge(START, "agent")
builder.add_conditional_edges("agent", route_after_agent,
    {"tools": "tools", "approval": "approval", END: END})
builder.add_conditional_edges("approval", route_after_approval,
    {"tools": "tools", "agent": "agent"})
builder.add_edge("tools", "agent")
graph = builder.compile(checkpointer=InMemorySaver())

tools_condition 查看最后一条 AI 消息：有 tool_calls 返回 "tools"，否则返回 END。ToolNode 会按 tool_call id 生成配对的 ToolMessage，并支持工具返回 Command 来更新状态。

6.7 人工审批：interrupt 与 resume

开启 AGENT_REQUIRE_APPROVAL 后，敏感工具先进入 approval 节点，用 interrupt 暂停并把待执行动作交给外部确认；恢复时用 Command(resume=...) 把答复送回同一位置。

agent.py · 审批节点
from langgraph.types import interrupt

def approval_node(state):
    last = state["messages"][-1]
    pending = [{"name": tc["name"], "args": tc["args"]}
               for tc in last.tool_calls if tc["name"] in SENSITIVE_TOOLS]
    decision = interrupt({"action": "confirm_sensitive_tools", "pending": pending})
    if str(decision).strip().lower() in {"y", "yes", "同意"}:
        return {"approved": True}
    denials = [ToolMessage("用户拒绝执行该操作。", tool_call_id=tc["id"])
               for tc in last.tool_calls]   # 必须回应每个 tool_call
    return {"approved": False, "messages": denials}

拒绝时要为“每一个”待处理 tool_call 生成一条 ToolMessage，否则消息序列不完整、模型下一次调用会报错。中断要求图启用 checkpointer，并使用相同 thread 配置恢复。CLI 侧的循环把中断展示给用户并回传答复：

agent.py · CLI 处理中断
while (payload := _pending_interrupt(graph, config)) is not None:
    print("[需要审批]", payload["pending"])
    answer = input("允许执行？(y/N)> ").strip() or "n"
    final = _stream_until_pause(graph, Command(resume=answer), config)

6.8 记忆：thread_id 与 checkpointer

带会话检查点的调用
config = {"configurable": {"thread_id": "demo"}}
graph.invoke({"messages": [HumanMessage(content="先看项目结构")]}, config)
# 同一 thread_id 的下一次调用会自动加载已有状态

InMemorySaver 只适合演示，进程退出即丢失。单机可换 SQLite checkpointer，生产环境常用 Postgres 等持久化实现。持久化含用户消息、代码片段与工具结果，应设置加密、保留期与访问控制。

6.9 终止、递归上限与无限循环

图的正常终止条件是模型不再发工具调用。但模型可能反复尝试失败的工具，因此要设置递归上限：

限制图步数
config = {
    "configurable": {"thread_id": "demo"},
    "recursion_limit": 40,
}

workspace/stringkit.py 的 truncate 截断时直接拼接后缀，导致结果长度超过 max_length；对应测试因此失败：

stringkit.py（有 bug）
def truncate(text: str, max_length: int, suffix: str = "...") -> str:
    if max_length < 0:
        raise ValueError("max_length must be non-negative")
    if len(text) <= max_length:
        return text
    # BUG: 没为后缀留空间，结果会比 max_length 更长
    return text[:max_length] + suffix

7.5 多轮继续与状态

保持相同 thread_id，可继续输入“再为 normalize_spaces 补一个只含空白的边界测试”。/state 查看已保存消息数、改动文件与执行命令；/new 切换会话；/quit 退出。

调用成本提示：同一会话消息会持续增长，每次模型调用可能重复传输历史。长任务应做历史裁剪或摘要，把大文件内容放外部存储并只在状态里保存引用。

常见失败的定位顺序

  1. 确认环境变量存在，但不要打印其值。
  2. 确认模型支持工具调用，模型名与账号权限正确。
  3. 查看最后一条 AIMessage 是否带 tool_calls。
  4. 查看 ToolMessage 的 tool_call_id 是否配对。
  5. 区分工具返回的“业务失败”与 Python 异常。
  6. 检查图的 next 节点、递归上限与同一 thread 的历史。

10. 常见问题

安装时报错要编译 tiktoken / ormsgpack，怎么办？

多半是在用 32 位 Python，缺少预编译 wheel 才会尝试用 Rust 从源码构建。改用 64 位 Python 3.10+ 重新建虚拟环境即可；或安装 Rust 工具链后再构建（不推荐）。

为什么不直接用 create_agent？

create_agent 是 LangChain 1.x 推荐的高层入口，也跑在 LangGraph 上。本文的目标是讲清流程控制，因此手工搭图。只需标准“模型—工具”循环时用它更省代码；需要自定义节点、审批、复杂路由或恢复时再下沉到 StateGraph。

工具为什么要返回 Command 而不是字符串？

返回字符串时 ToolNode 会自动包成 ToolMessage，适合只读工具。若还要更新自定义状态字段（如 changed_files），就返回 Command(update=...)，用 InjectedToolCallId 生成配对的 ToolMessage。

如何换成其他外部模型？

安装对应厂商的 LangChain 集成，替换 ChatOpenAI 初始化，保留 bind_tools 与图结构即可。必须确认模型支持工具调用，并改用该厂商的 Key 环境变量。仅当端点确实兼容官方 OpenAI 协议时才使用 OPENAI_BASE_URL。

InMemorySaver 是“长期记忆”吗？

不是。它是进程内 thread 级检查点，重启即丢。跨会话的用户记忆通常用 LangGraph Store；可恢复执行则用持久化 checkpointer，两者用途不同。

怎样让工具执行失败后自动重试？

先区分暂时错误与永久错误。网络超时可在节点用 RetryPolicy 或客户端有限指数退避；参数错误、权限拒绝不应盲目重试。写入与命令等有副作用的动作必须考虑重放与幂等。
