1. Agent 与 LangGraph 到底解决什么

普通 LLM 调用是“一问一答”；Agent 则让模型在一个受控循环里反复观察状态、选择工具、执行动作、读取结果，直到得到最终答案。LangGraph 把这个循环显式建模为图。

S

State 状态

消息、任务进度、工具结果等在节点间传递的数据。状态是图唯一可信的“共享白板”。

N

Node 节点

执行一次确定职责：调用模型、运行工具、审批、总结或写数据库。

E

Edge 边

决定下一步去哪里。条件边让工具调用回到模型，让普通回复走向结束。

什么时候值得使用 LangGraph

场景建议原因
单次摘要、翻译、分类直接调用模型没有循环、状态恢复或工具编排需求。
标准“模型 ↔ 工具”Agent可先用 LangChain create_agentLangChain 1.x 的高层 Agent 本身运行在 LangGraph 上，样板代码更少。
自定义审批、重试、分支、并行或恢复使用 LangGraph需要明确控制流程、状态和持久化边界。
代码修改、数据处理、长任务使用 LangGraph + 隔离工具可观察、可恢复、可审计，并能把风险动作放入审批节点。
版本提醒：LangGraph 1.x 已弃用 langgraph.prebuilt.create_react_agent，标准高层入口迁移为 langchain.agents.create_agent。本文为讲清底层流程，直接使用稳定的 StateGraph、ToolNode 和条件边。

2. 最重要的心智模型：受控的 ReAct 循环

Model分析状态并决定是否调用工具
→
Router检查 AIMessage.tool_calls
→
ToolNode执行函数并产生 ToolMessage
有工具调用：ToolNode → Model；无工具调用：Model → END

一次 Agent 执行不是模型在内部“神秘地做完所有事”，而是多轮可见的状态转换：

  1. 用户消息进入 MessagesState。
  2. 模型读取系统提示和消息历史，返回普通文本或结构化工具调用。
  3. tools_condition 判断是否存在工具调用。
  4. ToolNode 执行对应 Python 函数，把结果追加为 ToolMessage。
  5. 模型根据工具结果继续计划；直到返回不含工具调用的最终消息。
  6. 检查点在超步边界保存状态，后续可通过同一 thread_id 继续。
关键原则：模型只负责“决定”，工具负责“执行”，图负责“约束流程”，状态负责“保存事实”。不要让模型文本本身承担真实世界副作用。

3. 编写 Agent 的完整流程

把目标写成可验收的任务

先写输入、输出和完成条件。例如：“读取 Python 项目，按用户要求修改代码，运行测试，最后报告变更与测试结果”。如果无法客观判断完成，Agent 也难以可靠停止。

划定权限和风险边界

明确可读目录、可写目录、可运行命令、联网权限、超时、输出大小、敏感文件和必须人工确认的动作。安全边界必须写在工具代码里，不能只写提示词。

设计状态 Schema

最小 Agent 可直接用 MessagesState。若还要记录计划、重试次数、审批结果、产物路径，可定义 TypedDict 并给累积字段配置 reducer。

把能力做成窄而清晰的工具

每个工具只做一件事，类型和 docstring 要准确；模型依靠它们理解什么时候调用。对路径、参数、返回长度和异常做验证。

初始化外部模型并绑定工具

密钥来自环境变量。选择支持 tool calling 的模型，通过 bind_tools 把 Python 工具 Schema 交给模型。

实现节点

模型节点返回状态增量，例如 {"messages": [response]}；工具节点可使用 LangGraph 的 ToolNode，它负责并行工具调用和 ToolMessage 对齐。

连边并编译图

从 START 到模型；模型经条件边去工具或 END；工具执行后回模型。编译时注入 checkpointer。

流式运行与可观察性

开发期查看每个状态更新和工具调用；面向用户时只输出合适的进度与最终答案。记录延迟、token、工具错误、循环次数和任务结果。

用故障案例验证

至少测试路径穿越、超大文件、二进制文件、命令超时、模型超时、工具异常、测试失败、重复调用、对话恢复和最大循环限制。

6. 关键实现逐段解析

完整实现见 langgraph_code_agent/agent.py。下面拆开最影响正确性与安全性的部分。

6.1 状态：何时用 MessagesState

MessagesState 内置 messages 字段及消息追加 reducer，节点只需返回新消息，不必手动拼接历史。需要业务字段时可扩展：

自定义状态示例
from typing_extensions import TypedDict
from typing import Annotated
from operator import add
from langgraph.graph.message import add_messages

class AgentState(TypedDict):
    messages: Annotated[list, add_messages]
    changed_files: Annotated[list[str], add]
    retry_count: int

Reducer 很关键：同一字段有多个节点写入时，未配置 reducer 通常是覆盖；消息或文件清单这类累积值应显式定义合并方式。

6.2 工具：真正的权限边界

路径限定（摘自 agent.py）
def safe_path(relative_path: str) -> Path:
    candidate = (root / relative_path).resolve()
    if candidate != root and root not in candidate.parents:
        raise ValueError("拒绝访问工作区之外的路径")
    if _is_sensitive(candidate):
        raise ValueError("拒绝访问可能包含密钥或凭据的文件")
    return candidate
  • 先 resolve() 再检查父目录，可阻止 ../ 和多数符号链接逃逸。
  • 读文件限制大小、编码和返回长度，避免上下文被巨型文件撑满。
  • 搜索限制匹配数；列表限制文件数。
  • 敏感文件同时在读取和写入入口拦截。
  • 命令使用参数数组及 shell=False，再检查程序白名单和超时。
白名单 ≠ 沙箱。允许 python 就意味着被运行的 Python 文件仍可能执行任意系统调用。真实生产环境要叠加容器、低权限用户、只挂载临时工作区、网络策略、资源配额和人工审批。

6.3 外部模型：读取 API Key 并绑定工具

模型初始化
if not os.getenv("OPENAI_API_KEY"):
    raise RuntimeError("未设置 OPENAI_API_KEY")

model = ChatOpenAI(
    model=os.getenv("OPENAI_MODEL", "gpt-5-mini"),
    timeout=60,
    max_retries=2,
)
model_with_tools = model.bind_tools(tools)

ChatOpenAI 自动读取 OPENAI_API_KEY。示例还支持可选的 OPENAI_BASE_URL，但兼容端点必须正确实现 OpenAI 工具调用协议；若厂商有专用 LangChain 集成，应优先使用厂商集成。

6.4 模型节点：输入完整状态，输出状态增量

模型节点
def call_model(state: MessagesState):
    response = model_with_tools.invoke([
        SystemMessage(content=SYSTEM_PROMPT),
        *state["messages"],
    ])
    return {"messages": [response]}

系统提示在每次模型调用时作为首条消息注入，不需要写入持久化历史。返回的列表只包含本轮新消息，MessagesState 会负责追加。

6.5 图：把循环写成明确的边

StateGraph 核心
builder = StateGraph(MessagesState)
builder.add_node("model", call_model)
builder.add_node("tools", ToolNode(tools, handle_tool_errors=True))

builder.add_edge(START, "model")
builder.add_conditional_edges(
    "model",
    tools_condition,
    {"tools": "tools", END: END},
)
builder.add_edge("tools", "model")

graph = builder.compile(checkpointer=InMemorySaver())

tools_condition 查看最后一条 AI 消息：有 tool_calls 时返回 "tools"，否则返回 END。ToolNode 会按 tool_call id 产生配对的 ToolMessage。

6.6 记忆：thread_id 与 checkpointer

带会话检查点的调用
config = {"configurable": {"thread_id": "demo-thread"}}
graph.invoke(
    {"messages": [HumanMessage(content="先阅读项目结构")]},
    config,
)
# 使用同一个 thread_id 的下一次调用会加载已有消息状态

InMemorySaver 只适合演示：进程退出即丢失。单机可换 SQLite checkpointer，生产环境通常使用 Postgres 等持久化实现。持久化含用户消息、代码片段和工具结果，应设置加密、保留期与访问控制。

6.7 终止、重试与无限循环

图的正常终止条件是模型不再发工具调用。但模型可能反复尝试失败工具。调用时可设置递归上限，并对可恢复错误与永久错误采用不同策略：

限制图步数
config = {
    "configurable": {"thread_id": "demo"},
    "recursion_limit": 30,
}

values 给出每步后的完整状态，适合简单 CLI；updates 给出节点更新，适合调试和事件 UI。生产日志中要脱敏，不应原样记录 API Key、凭据文件或用户私有代码。

常见失败的定位顺序

  1. 确认环境变量存在，但不要打印值。
  2. 确认模型支持工具调用，模型名和账号权限正确。
  3. 查看最后一条 AIMessage 是否有 tool_calls。
  4. 查看 ToolMessage 的 tool_call_id 是否匹配。
  5. 区分工具返回的业务失败与 Python 异常。
  6. 检查图的 next 节点、递归上限和同一 thread 的历史。

9. 从演示到生产：需要补齐什么

领域演示做法生产建议
隔离路径校验 + 命令白名单每任务短生命周期容器/沙箱、低权限用户、只挂载工作区、默认禁网、CPU/内存/进程限制

人工审批节点的基本形态

概念示例
from langgraph.types import interrupt

def approval_node(state):
    decision = interrupt({
        "question": "是否允许执行以下写入？",
        "proposed_changes": state["pending_changes"],
    })
    return {"approved": decision == "yes"}

中断要求图启用 checkpointer；恢复时使用相同 thread 配置和 Command(resume=...)。审批内容要展示确定的目标、diff、命令和影响范围，而不是笼统问“是否继续”。

上线前检查清单

  • 工具参数全都有类型、描述、边界验证和确定的错误返回。
  • 图有明确成功条件、失败条件、超时、最大步数和取消路径。
  • 任何不可逆或外部副作用都有审批与幂等保护。
  • 持久化状态不包含不必要的密钥、凭据或超大数据。
  • 真实验收基于测试、文件 diff 或结构化结果，而不是模型自称“完成”。
  • 成本、速率、并发和日志保留均有上限。
  • 针对仓库内恶意提示、依赖脚本与测试脚本做过威胁建模。

10. 常见问题

为什么不直接使用 create_agent？

create_agent 是 LangChain 1.x 推荐的高层入口，也运行在 LangGraph 上。本文的目标是解释流程控制，因此手工搭建图。若只需标准模型—工具循环，create_agent 更省代码；需要自定义节点、审批、复杂路由或恢复时再下沉到 StateGraph。

API Key 能作为参数直接传给 ChatOpenAI 吗？

技术上可以，但应用代码中硬编码非常容易泄漏。环境变量或 Secret Manager 更适合。也不要把 Key 作为用户消息放入图状态，因为检查点和日志可能持久化它。

如何换成其他外部模型？

安装对应厂商的 LangChain 集成，替换 ChatOpenAI 初始化，保留 bind_tools 和图结构即可。必须确认模型支持工具调用，并改用该厂商的 Key 环境变量。仅当端点确实兼容官方 OpenAI 协议时才使用 OPENAI_BASE_URL。

为什么 Agent 没有调用工具？

检查模型是否支持 tool calling、工具 docstring 是否清晰、用户任务是否真的要求动作、工具是否正确 bind。不要靠提示词强迫每轮调用；最终总结本就不应调用工具。

InMemorySaver 是“长期记忆”吗？

不是。它是进程内 thread 级检查点，重启即丢失。跨会话的用户记忆通常使用 LangGraph Store；可恢复执行则使用持久化 checkpointer，两者用途不同。

怎样让工具执行失败后自动重试？

先区分暂时错误与永久错误。网络超时可在节点使用 RetryPolicy 或在客户端做有限指数退避；参数错误、权限拒绝不应盲目重试。命令与写入等有副作用的动作必须考虑重放和幂等。
